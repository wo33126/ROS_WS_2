## 1. 系统启动

### 1.1 完整任务模式（推荐比赛使用）

```bash
# 启动所有节点（完整任务流程）
roslaunch robot_navigation mission_complete.launch

# 可选参数
roslaunch robot_navigation mission_complete.launch \
    enable_camera:=true \
    enable_arm:=true \
    enable_servo:=true
```

启动后各节点按依赖顺序加载：

| 层级 | 节点 | 功能 |
|:---:|:---|:---|
| 1 | `can_interface_node` | 底盘 CAN 通信 |
| 2 | `cmd_vel_mux_node` + `base_odometry_node` + `planner_node` | 运动规划 |
| 3 | `path_manager_node` + `path_tracker_node` | 路径导航 |
| 4 | `fine_tuning_node` | 终点 VL53L1X 微调 |
| 5 | `servo_controller_node` + `arm_and_gripper_node` | 药品摆放 |
| 6 | `barcode_qr_detector_node` + `open_medicine_box_node` | 视觉+药箱 |
| 7 | `display_node` + `start_signal_pub_node` + `start_button_node` | 人机交互 |
| 7 | `video_stream_node` + `audio_chat_node` | 音视频 |
| 8 | `vl53l1x_driver_node` + `distance_safety_node` | 安全感知 |
| 8 | `emergency_stop_node` | 急停 |
| 9 | `speak_node` + `health_monitor_node` + `mission_controller_node` | 任务编排 |

### 1.2 动态导航模式（含 AMCL + move_base）

```bash
# 需要先构建地图（首次使用）
roslaunch robot_navigation dynamic_navigation.launch
```

### 1.3 基础调试模式

```bash
# 仅底盘 + 运动规划，不启动任务状态机
roslaunch robot_navigation complete_navigation.launch \
    enable_teleop:=true
```

## 3. 音视频通信

> **对应赛事规则**：医生与病人通过机器人显示屏进行音视频交流（1床+3床各20分，共40分）  
> **关键要求**：音频交流的同时必须能视频，否则不得分。

### 3.1 视频推流（机器人端 → 医生端）

机器人端自动启动 MJPEG 视频流，医生端通过浏览器查看：

```
http://<机器人IP>:8080/stream
```

**配置参数**（`navigation_params.yaml`）：

```yaml
video_stream:
  http_port: 8080          # HTTP 服务端口
  jpeg_quality: 70         # JPEG 压缩质量 (1-100)
  stream_width: 640        # 视频宽度
  stream_height: 480       # 视频高度
  framerate: 15.0          # 推流帧率
```

**医生端操作**：
1. 确保医生端 PC 与机器人处于同一网络
2. 打开浏览器访问 `http://<机器人IP>:8080/stream`
3. 即可看到机器人摄像头实时画面（MJPEG 格式，支持所有主流浏览器）

**摄像头要求**：
- USB 摄像头（UVC 协议），`/dev/video0`
- 由 `usb_cam` 包驱动，发布 `/usb_cam/image_raw`

### 3.2 双向音频（机器人端 ↔ 医生端）

**配置参数**：

```yaml
audio_chat:
  enabled: true
  sample_rate: 16000       # 采样率 16kHz
  doctor_ip: "192.168.1.100"  # ⚠️ 修改为医生端实际 IP
  rx_port: 5004            # 机器人接收端口
  tx_port: 5005            # 机器人发送端口
```

**使用步骤**：
1. **修改 `doctor_ip`** 为医生端 PC 的实际 IP 地址
2. 发布空消息到 `/audio_chat/control` 切换音频开关：
   ```bash
   rostopic pub /audio_chat/control std_msgs/Empty "{}"
   ```
3. 查看音频状态：
   ```bash
   rostopic echo /audio_chat/active
   ```

**医生端需运行对应的音频收发程序**（见附录 A）。

**硬件要求**：
- 机器人端：USB 麦克风 + 扬声器（3.5mm 或 USB）
- 医生端：麦克风 + 扬声器

**语音播报**：除双向音频外，`speak_node` 提供独立的 TTS 播报（espeak-ng），
在比赛中由 `mission_controller` 自动调用（到达病床放药后），无需手动操作。

## 4. 物理按钮操作

> **对应赛事规则**：只允许通过机器人车体上的按键操作启动，不允许键盘/鼠标/触摸屏/遥控器。

### 4.1 启动按钮

**硬件接线**（Raspberry Pi）：

```
按钮一端 → GPIO 17 (BCM, 物理引脚 11)
按钮另一端 → GND
LED(+) → GPIO 27 (BCM, 物理引脚 13) → 220Ω 电阻 → GND
```

**行为说明**：
| 状态 | LED 指示灯 | 说明 |
|:---|:---:|:---|
| 就绪等待 | 常亮 | 系统已启动，等待按下 |
| 按下触发 | 灭 1 秒后恢复常亮 | 启动信号已发送 |
| 任务执行中 | 常亮 | 任务状态机运行中 |

**发布话题**：
- `/start_signal`（std_msgs/Empty, latch=true）— 任务状态机监听此话题
- `/start_signal_manual`（std_msgs/Empty）— 备用调试话题

**去抖动**：50ms 去抖，防止误触发。

**手动测试**（不按物理按钮）：
```bash
rostopic pub /start_signal std_msgs/Empty "{}"
```

### 4.2 启动流程

```
┌─────────────┐    按下按钮     ┌──────────────┐
│ IDLE (等待) │ ──────────────→ │ GOTO_NURSE   │ → 开始完整任务
└─────────────┘                 └──────────────┘
```

**注意**：
- 按钮按下后任务立即开始，无二次确认
- 任务进行中再次按下按钮 **不会** 中断当前任务
- 只有任务完成/失败/超时后，按钮才可再次启动新任务

## 5. 急停按钮

> **对应赛事规则**：机器人应设有急停按钮。

### 5.1 硬件接线

```
急停按钮 NC（常闭）→ GPIO 22 (BCM, 物理引脚 15)
急停按钮 COM → GND
急停指示灯 LED(+) → GPIO 23 (BCM, 物理引脚 16) → 220Ω → GND
```

⚠️ **强烈建议**：急停按钮的常闭触点同时串联到电机电源继电器，实现 **硬件级断电保护**。

### 5.2 双重保护机制

| 层级 | 保护方式 | 触发时行为 |
|:---|:---|:---|
| 软件层 | GPIO 检测 → `/emergency_stop` → `cmd_vel_mux` 强制输出零速度 | 电机指令立即清零 |
| 硬件层（建议） | 常闭触点串联电机电源 | 电机物理断电 |

### 5.3 行为说明

| 状态 | 指示灯 | 说明 |
|:---|:---:|:---|
| 正常运行 | 常亮 | 急停未触发 |
| 急停按下 | 灭 | 机器人立即停止 |
| 急停释放 | 恢复常亮 | 需重新按启动按钮开始任务 |

**发布话题**：
- `/emergency_stop`（std_msgs/Bool, latch=true）— `cmd_vel_mux` 监听
- `/emergency_stop/state`（std_msgs/Bool）— 急停状态
- `/emergency_stop/event`（std_msgs/Empty）— 急停事件（每次按下发布一次）

**手动测试**：
```bash
# 触发急停
rostopic pub /emergency_stop std_msgs/Bool "data: true"

# 释放急停
rostopic pub /emergency_stop std_msgs/Bool "data: false"
```

## 6. 动态避障 (Bug2)

> **对应赛事风险**：黄色区域内随机放置障碍物（500×500×500mm），碰撞每次扣5分。

### 6.1 工作原理

```
正常行驶                前方障碍 < 300mm
    │                       │
    ▼                       ▼
  CLEAR ──────────→ STOP（停止）──→ Bug2 激活
                                        │
                          ┌─────────────┴─────────────┐
                          ▼                           ▼
                    检查左侧空间                  检查右侧空间
                          │                           │
                          └───────────┬───────────────┘
                                      ▼
                              转向空间大的一侧 ~90°
                                      │
                                      ▼
                              沿障碍边缘前进
                                      │
                                      ▼
                              回归目标方向 → CLEAR
```

### 6.2 安全距离等级

| 距离 (mm) | 状态 | 行为 |
|:---:|:---:|:---|
| > 450 | CLEAR | 正常行驶 |
| 300 ~ 450 | WARNING | 减速至 30% |
| 100 ~ 300 | STOP | 立即停止 + Bug2 绕障 |
| < 100 | REVERSE | 紧急后退 |

### 6.3 配置参数

```yaml
distance_safety:
  safety_distance_mm: 300.0       # 安全停止距离
  critical_distance_mm: 100.0     # 紧急后退距离
  reverse_speed: 0.05             # 后退速度 (m/s)
  check_forward_only: true        # 仅检测前方
  bug2_enabled: true              # 启用 Bug2 绕障
  bug2_side_clearance_mm: 400.0   # 侧向通过所需最小空间
  bug2_turn_speed: 0.5            # 绕障转向速度 (rad/s)
  bug2_along_speed: 0.1           # 绕障前进速度 (m/s)
  bug2_max_duration_sec: 10.0     # 最大绕障时间（超时放弃）
```

### 6.4 侧向传感器（可选，推荐）

如果需要更可靠的 Bug2 绕障，建议安装左右两个 VL53L1X：

| 传感器 | I2C 地址 | 发布话题 |
|:---|:---:|:---|
| 前方 | 0x29 | `/vl53l1x_distance` |
| 左侧 | 0x30 | `/vl53l1x_distance_left` |
| 右侧 | 0x31 | `/vl53l1x_distance_right` |

**注意**：多个 VL53L1X 在同一 I2C 总线上需要将 XSHUT 引脚分别接到不同的 GPIO，
逐个上电并分配不同地址。详情参考 VL53L1X 数据手册。

**无侧向传感器时**：Bug2 默认假设侧向空间足够（400mm），仍会尝试绕障。

## 7. 条码持久显示

> **对应赛事规则**：1床和3床条码数值正确显示后保留一直显示，比赛结束裁判核对。  
> 如果需要操作按键切换界面调出条码数值的不得分（10分）。

### 7.1 显示格式

机器人显示屏上始终保持如下格式：

```
1床条码: 12345678
3床条码: 87654321
```

- 第 1 行：1 床条码（扫描后锁定，不再更新）
- 第 2 行：3 床条码（扫描后锁定，不再更新）
- 任务结束后追加 "--- 任务完成 ---"

### 7.2 实现原理

- 条码值由 `mission_controller` 保存在 `barcode_display_line1_` 和 `barcode_display_line2_` 中
- 任务重置时 **不会** 清除这两行（直到系统重启或手动清除）
- 显示通过 `/robot_display` 话题发送给 `display_node`
- `display_node` 输出到 ROS 日志 + 可选 CAN 液晶屏

### 7.3 手动查看显示内容

```bash
# 查看当前显示内容
rostopic echo /robot_display

# 手动设置显示内容（调试用）
rostopic pub /robot_display std_msgs/String "data: '1床条码: 12345678\n3床条码: 87654321'"
```

## 8. 底盘锁死机制

> **对应赛事规则**：药箱开启到语音播报结束，底盘必须全部在半径 300mm 圆圈内且不可移动。  
> 移动扣 5 分，不完全在圈内扣 5 分。

### 8.1 触发时机

```
OPEN_BOX → PLACE_MEDICINE → VOICE
   🔒           🔒           🔒 → 🔓
```

- **`lockChassis()`**：进入 `OPEN_BOX` 状态时调用
- 保持锁死贯穿 `OPEN_BOX` → `PLACE_MEDICINE` → `VOICE`
- **`unlockChassis()`**：语音播报完成后调用

### 8.2 锁死操作

1. 发布 `std_msgs/Bool(true)` 到 `/chassis_lock`
2. 发布 `std_msgs/Empty` 到 `/stop_all` 确保立即停止
3. 建议：在 `cmd_vel_mux` 中监听 `/chassis_lock`，锁定时拒绝所有非零速度指令

### 8.3 监控方式

```bash
# 查看底盘锁状态
rostopic echo /chassis_lock

# 手动锁定（调试用）
rostopic pub /chassis_lock std_msgs/Bool "data: true"

# 手动解锁
rostopic pub /chassis_lock std_msgs/Bool "data: false"
```

## 9. 语音播报时机

> **对应赛事规则**：药品摆放之后 5 秒内语音播报得 5 分，其他时间播报不得分。

### 9.1 计时机制

- 药品放置完成时记录 `medicine_placed_time_`
- 进入 `VOICE` 状态时校验：`当前时间 - 放置完成时间 ≤ 5.0 秒`
- 若在 5 秒内：日志输出 "✅ 语音播报在 X.X 秒内（5秒内有效）"
- 若超 5 秒：日志输出 "⚠ 语音播报超时（不得分但继续流程）"

### 9.2 优化建议

如果经常超 5 秒，检查以下环节：
1. `arm_and_gripper_node` 的放置动作是否耗时过长
2. 放置完成信号（服务返回）是否有延迟
3. 调整 `callSpeak` 的优先级，确保语音播报在放置动作后立即执行

## 10. 动态导航 (AMCL + move_base)

> **对应赛事需求**：提高对随机障碍物的适应性，替代纯固定路线。

### 10.1 前置条件

```bash
# 安装依赖包
sudo apt install ros-noetic-move-base ros-noetic-amcl ros-noetic-teb-local-planner ros-noetic-map-server

# 构建场地地图（需要激光雷达或深度相机）
# 1. 手动遥控机器人扫描场地
# 2. 使用 gmapping 或 hector_slam 构建地图
# 3. 保存地图到 robot_navigation/maps/arena.yaml
```

### 10.2 启动

```bash
# 动态导航模式
roslaunch robot_navigation dynamic_navigation.launch

# 加载指定地图
roslaunch robot_navigation dynamic_navigation.launch map_file:=/path/to/map.yaml
```

### 10.3 与固定路线模式对比

| 特性 | 固定路线 (path_tracker) | 动态导航 (move_base) |
|:---|:---|:---|
| 路径来源 | YAML 预设路径点 | 全局代价地图实时规划 |
| 避障能力 | 仅 stop | 局部规划器动态绕行 |
| 定位方式 | 里程计推算 | AMCL 概率定位 |
| 适用场景 | 无障碍赛道 | 有随机障碍物赛道 |
| CPU 消耗 | 低 | 中等 |
| 推荐使用 | 训练/测试 | 正式比赛 |

### 10.4 两种模式共存

系统支持两种模式共存：固定路线作为 fallback，动态导航作为首选。

在 `mission_controller` 中可通过参数切换导航方式（后续版本功能）。

## 11. 任务超时与错误恢复

### 11.1 三级超时保护

| 级别 | 时间 | 触发条件 | 恢复策略 |
|:---|:---:|:---|:---|
| 阶段超时（首次） | 30s | 单个状态耗时过长 | 跳过当前阶段，进入下一阶段 |
| 阶段超时（二次） | 30s | 连续两阶段超时 | 进入失败状态 |
| 全局超时 | 180s | 整个任务超时 | 跳过当前任务，直接返回起始区 |

### 11.2 可跳过的阶段

以下非关键阶段超时后会被跳过：

| 阶段 | 跳过后进入 |
|:---|:---|
| SCAN_QR | GOTO_BED_A（QR 为空则后续可能失败） |
| POSITION_IN_CIRCLE | SCAN_BARCODE（微调失败不致命） |
| SCAN_BARCODE | OPEN_BOX（无条码仍可继续） |
| HOME_CHECK | STOP（直接结束） |

### 11.3 不可跳过的阶段

以下关键阶段超时后会直接进入失败：
- `GOTO_NURSE`、`GOTO_BED_A`、`GOTO_BED_B`、`RETURN_HOME`（导航阶段）
- `OPEN_BOX`、`PLACE_MEDICINE`（同步完成，不会超时）

### 11.4 全局超时恢复

全局 180 秒超时后，系统不会立即失败，而是：
1. 解锁底盘
2. 直接进入 `RETURN_HOME` 状态
3. 给回归分配 60 秒时间
4. 回归成功仍可得回归分（20 分）

## 12. 配置文件参考

### 12.1 主要配置文件

| 文件 | 用途 |
|:---|:---|
| `robot_navigation/config/navigation_params.yaml` | 所有导航相关参数（主配置） |
| `robot_bringup/config/robot_params.yaml` | 机器人物理参数 |
| `robot_bringup/config/paths.yaml` | 路径点定义 |
| `robot_bringup/config/path_tracker.yaml` | 路径跟踪参数 |
| `fine_tuning/config/fine_tuning.yaml` | 终点微调参数 |
| `arm_and_gripper/config/arm_and_gripper.yaml` | 机械臂参数 |

### 12.2 任务状态机参数

```yaml
mission_controller:
  stage_timeout_sec: 30.0        # 单阶段超时（秒）
  mission_timeout_sec: 180.0     # 全局超时（秒，比赛3分钟限制）
  state_machine_rate_hz: 10.0    # 状态机循环频率
  home_hold_sec: 5.0             # 回到起始区稳定时间

  # 病床圆圈位置（坐标系原点为起始区中心，前方为+X）
  circle_bed1_x: 2.0             # 1床圆圈中心 X
  circle_bed1_y: 1.0             # 1床圆圈中心 Y
  circle_bed1_radius: 0.3        # 1床圆圈半径（300mm）
  circle_bed3_x: 2.0
  circle_bed3_y: -1.0
  circle_bed3_radius: 0.3

  # 起始区多边形顶点
  home_zone_vertices:
    - {x: -0.5, y: -0.5}
    - {x:  0.5, y: -0.5}
    - {x:  0.5, y:  0.5}
    - {x: -0.5, y:  0.5}
```

### 12.3 医生端 IP 配置

## 附录 A：医生端音频程序

以下为医生端简易音频收发 Python 脚本（保存为 `doctor_audio.py`）：

```python
#!/usr/bin/env python3
""" 医生端音频收发 — 与机器人 audio_chat_node 配合使用 """

import socket
import subprocess
import threading
import sys

ROBOT_IP = "192.168.1.50"   # ← 改为机器人实际 IP
RX_PORT = 5005              # 接收端口（对应机器人 TX）
TX_PORT = 5004              # 发送端口（对应机器人 RX）

def receive_audio():
    """接收机器人端音频并播放"""
    sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    sock.bind(("0.0.0.0", RX_PORT))
    sock.settimeout(0.1)

    player = subprocess.Popen(
        ["aplay", "-f", "S16_LE", "-r", "16000", "-c", "1", "-t", "raw"],
        stdin=subprocess.PIPE
    )

    print(f"[RX] 监听端口 {RX_PORT}...")
    while True:
        try:
            data, addr = sock.recvfrom(4096)
            player.stdin.write(data)
            player.stdin.flush()
        except socket.timeout:
            continue
        except BrokenPipeError:
            break

def send_audio():
    """采集医生端麦克风并发送到机器人"""
    sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)

    recorder = subprocess.Popen(
        ["arecord", "-f", "S16_LE", "-r", "16000", "-c", "1", "-t", "raw"],
        stdout=subprocess.PIPE
    )

    print(f"[TX] 发送到 {ROBOT_IP}:{TX_PORT} ...")
    while True:
        data = recorder.stdout.read(640)  # 20ms × 16kHz × 2bytes
        if not data:
            break
        sock.sendto(data, (ROBOT_IP, TX_PORT))

if __name__ == "__main__":
    if len(sys.argv) > 1:
        ROBOT_IP = sys.argv[1]

    print(f"医生端音频: 机器人={ROBOT_IP}, RX={RX_PORT}, TX={TX_PORT}")
    threading.Thread(target=receive_audio, daemon=True).start()
    send_audio()
```

使用方法：
```bash
python3 doctor_audio.py <机器人IP>
```
